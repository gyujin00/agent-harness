# PRD — SRM MVP (공급업체 등록 관리 / 평가·성과 관리 / 대시보드)

## 목적

- 배경/문제: 구매팀은 약 80개 공급업체 정보가 담당자별 엑셀 파일로 분산 관리되어 최신성·일관성이
  보장되지 않고(P1), 동일 업체가 파일마다 다른 명칭으로 중복 등록되며(P2), 분기별 평가가 담당자마다
  다른 기준/양식으로 수행되어 업체 간·기간 간 비교가 불가능하고(P3), 경영진이 전체 현황(업체 수, 등급
  분포, 저성과 업체)을 파악하려면 담당자 수작업 취합이 필요하다(P4).
  [출처: `docs/domain-researcher/outputs/research-report.md` 페인포인트 P1~P4]
- 이 PRD가 정의하는 것: 위 문제를 해결하기 위한 SRM MVP 3종(공급업체 등록 관리, 평가·성과 관리,
  대시보드)의 요구사항.
- 대상 사용자: 구매팀 담당자(등록·검색·평가 수행), 구매팀 평가 담당자(평가 입력), 경영진/구매팀장
  (대시보드 열람). [출처: 페인포인트 P1~P4 "대상 사용자" 컬럼]

## 범위

- In: 공급업체 등록 관리, 평가·성과 관리, 현황 대시보드.
- Out (본 PRD는 다루지 않음): 계약/발주/정산, 실시간 ERP 연동, 공급망 리스크 모니터링, 외부 포털,
  실제 LLM API 기반 자동 의사결정, 실제 기업/협력사 데이터.
  [출처: 연구보고서 "목적" 절 범위 In/Out, 제약 C2·C3·C4]

## 우선순위 판단 근거

- 필수(must): REQ-001~006(등록 관리), REQ-007~010(평가·성과), REQ-011~013(대시보드 핵심 지표),

  (R1의 "관리 및 조회·수정" 중 "수정" 부분은 REQ-003으로 반영했다).
- 부가(should): REQ-014(AI 요약)은 연구보고서 "권장 범위"가 "대시보드 핵심 지표보다 후순위"로 명시
  권고했고, 실제 LLM 기반 의사결정과 혼동되지 않도록 규칙 기반 구현이 전제되어야 하는 제약(C3)이
  있어 후순위로 둔다.
- 부가(should): REQ-016~022(AI 보조 기능 5종 + REQ-021·022, REQ-016·020에서 분리된 폴백 요구사항)도
  REQ-014와 같은 이유로 부가(should)로 둔다 — 이들은 이미 필수로 확정된 핵심 흐름(등록/조회
  REQ-001~006, 평가 입력·자동 산출 REQ-007~009, 대시보드 핵심 지표 REQ-011~013)을 보완하는 부가
  기능이지 그 자체가 MVP 핵심 흐름을 구성하지 않는다. REQ-019(등급조정 초안)는 특히 `AGENTS.md` §5가
  금지하는 "실제 LLM API 기반 자동 의사결정"과 혼동되지 않도록 "사람이 승인하는 초안"으로 범위를
  제한했다는 점에서 REQ-014의 규칙 기반 제약(C3)과 같은 성격의 안전장치를 갖는다.
