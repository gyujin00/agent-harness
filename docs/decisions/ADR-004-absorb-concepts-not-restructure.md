# ADR-004: 이사님 번호폴더 구조를 전면 도입하지 않고 개념만 흡수한다

- 상태: accepted
- 날짜: 2026-07-17
- 도메인: harness
- 관련 루프/문서: plans/, harness/verification.policy.md, docs/traceability.md, docs/project-state.md

## 맥락
2차 미팅에서 이사님이 `00_overview ~ 10_outputs`의 번호 매긴 11개 폴더 구조를 권장했다.
이를 그대로 도입할지, 아니면 핵심 개념만 우리 2계층 구조(ADR-001)에 흡수할지 결정이 필요했다.

## 대안
- (A) 번호 폴더로 전면 재구조화 — 피드백 100% 반영, 문서 가독성↑. 그러나 기존 하네스 설계 상당 부분 재작업.
- (B) 하이브리드 — 2계층 위에 번호 폴더를 겹침. 절충이지만 구조가 이중이 되어 복잡.
- (C) 개념만 흡수 — 폴더 구조는 유지, 빠진 개념(Sprint/Task, 문서 검증, 추적성, project-state)만 추가.

## 결정
(C). 사용자 확인을 거쳐 개념만 흡수한다.

## 근거
- 이사님의 번호 폴더는 **문서화 중심(SI/산출물 관점)** 구조이고, 우리 것은 **실행 하네스 중심**이라
  철학이 달라 그대로 이식하면 두 계층 모델(ADR-001)과 충돌한다.
- 정작 피드백의 알맹이는 폴더 번호가 아니라 4가지 개념(계획 계층·문서 검증·추적성·현재 상태)이다.
  구조는 유지하고 알맹이만 취하는 편이 리스크 최소·가치 최대.

## 시행착오 / 검증
- 도입 전, 두 전사본(내용정리/클로바노트)을 공식 문서와 대조해 피드백을 4개 개념으로 압축.
- 사용자에게 도입 수준(개념 흡수 vs 전면 재구조화 vs 하이브리드)을 명시적으로 질문해 (C) 확정.
- 흡수 후 hook 10개 시나리오 회귀 테스트, 심볼릭 링크·frontmatter·agents 최신성 전부 재검증.

## 결과 / 영향
- 생성: `plans/`(Sprint·Task), `harness/verification.policy.md`, `docs/traceability.md`, `docs/project-state.md`.
- 변경: `AGENTS.md §3.5`, `permissions.policy.md`(orchestrator write 확장), hook POLICY 미러.
- 되돌리기 비용: 중간. 추가한 4개 개념은 독립적이라 개별 제거 가능하나 권한/문서 연결을 함께 정리해야 함.
