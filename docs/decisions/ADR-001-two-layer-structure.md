# ADR-001: 환경/실행 2계층으로 하네스를 분리한다

- 상태: accepted
- 날짜: 2026-07-17
- 도메인: harness
- 관련 루프/문서: AGENTS.md §0, README.md

## 맥락
하네스(무엇을 주고 어떻게 통제할지)와 루프(무엇을 반복할지)를 한 덩어리로 두면,
"통제하는 판"과 "반복하는 게임"이 섞여 사람이 전체 개발 체계를 한눈에 볼 수 없다.
2차 미팅에서 이사님이 지적한 핵심도 "좋은 구조가 에이전트 내부에 숨어 사람에게 안 보인다"였다.

## 대안
- (A) 단일 `CLAUDE.md`에 규칙·워크플로우를 전부 기술 — 파일이 비대해지고 규칙이 묻힘.
- (B) 도메인별 폴더(backend/frontend/ai)만 나누기 — 통제와 실행 구분이 여전히 안 보임.
- (C) 환경 계층(harness/·.claude/·eval/·docs/)과 실행 계층(plans/·loops/)을 명시적으로 분리.

## 결정
(C)를 택한다. 한 문장 원칙: **환경은 손대는 사람이 바꾸고, 실행은 plan과 loop.spec이 바꾼다.**

## 근거
- 하네스 엔지니어링의 원칙(정적 통제 판 / 동적 실행 게임 분리)에 부합.
- 평가 기준(과제 PDF §5·§6)이 "기능이 아니라 통제 체계·과정"을 본다 → 체계가 구조로 드러나야 함.
- 계층이 분리돼야 정책 변경(환경)과 일감 실행(실행)의 책임 소재가 갈린다.

## 시행착오 / 검증
- 처음엔 harness/와 loops/만 두었으나, Sprint/Task(계획)를 어디에 둘지 모호했다(ADR-004에서 plans/로 해소).
- AGENTS.md §0에 두 계층의 구성 요소를 명문화하고, 모든 후속 문서가 이 구분을 참조하는지 확인.

## 결과 / 영향
- 생성/변경: `AGENTS.md §0`, `README.md`(두 계층 절), 전체 폴더 배치의 기준이 됨.
- 영향: 이후 모든 ADR·문서가 "환경 vs 실행" 어느 계층인지 명시.
- 되돌리기 비용: 낮음(문서 재배치 수준). 단, 하위 문서가 이 구분에 의존하므로 변경 시 연쇄 갱신 필요.
