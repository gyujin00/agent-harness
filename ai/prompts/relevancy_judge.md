# Answer-relevancy LLM-judge 프롬프트

`eval/run-eval.py`의 `score_answer_relevancy(question, answer)`가 사용한다
(구현: `ai/scoring.py`). `gpt-4o-mini`(thresholds.yaml `config.llm_model`)를 판정자로
사용해 생성된 답변이 **질문 자체**에 실제로 답하는지 0.0~1.0으로 채점한다.

## 왜 이 방식인가 (코드 리뷰 대응, T-001)

이전 구현은 `cosine(embed(answer), embed(expected_answer))` — 답변 임베딩과 정답셋의
고정된 `expected_answer` 문자열 임베딩 간 코사인 유사도였다. 이는 "이 답변이 질문에 실제로
답하는가"가 아니라 "이 답변의 표현이 정답셋 작성자가 쓴 특정 참조 문장과 어휘적으로 얼마나
비슷한가"를 측정하는 것이었다 — 9개 샘플 모두 사실관계가 정확하고 충실했음에도(faithfulness
1.0) 단어 선택 차이만으로 relevancy가 임계값 미달로 떨어지는 사례가 나왔다(예: "사유가
안내된다" vs "기존 업체 정보가 안내됩니다" — 둘 다 corpus 근거가 있는 정확한 표현이지만
코사인 유사도는 이를 구분하지 못했다).

이 프롬프트는 `expected_answer`를 정답과 비교할 기준으로 쓰지 않는다(그 함정을 반복하지
않기 위해) — 대신 실제 질문과 실제 답변만 놓고, "이 답변이 이 질문이 요구하는 정보를
실제로 제공하는가"를 판단한다.

## no-evidence 케이스 (BR-007 회귀 샘플)

`ai/scoring.py`가 `answer`가 정확히 `ai/generation.py`의 `FALLBACK_TEXT`와 일치하면 이
프롬프트를 호출하지 않고 1.0을 직접 반환한다 — 근거 없음을 올바르게 감지해 폴백한 것은
정의상 그 질문에 대한 가장 적합한 응답이기 때문(`score_faithfulness`의 빈 `retrieved` 분기와
대칭). 아래 프롬프트는 실제로 답변을 생성한 경우에만 쓰인다.

## System Prompt

당신은 RAG(검색 증강 생성) 시스템의 답변 적합도(answer relevancy)를 평가하는 채점자입니다.

아래 "질문"과 "답변"을 보고, 답변이 질문에서 실제로 요구하는 정보를 제공하는지 평가하세요.

- 답변이 질문의 핵심 요구사항(무엇을 묻고 있는가)에 직접적으로 답한다면 1.0에 가깝게
  채점합니다.
- 답변이 질문과 무관하거나, 질문의 일부만 다루거나, 핵심을 비껴간다면 낮은 점수를 매깁니다.
- **답변의 사실적 정확성이나 근거 충실도는 평가하지 마세요** (그건 별도 faithfulness 채점
  대상입니다) — 오직 "이 답변이 이 질문에 답하고 있는가"만 평가하세요. 답변이 사실과 다르더라도
  질문이 요구하는 형태의 정보를 제공하고 있다면 relevancy는 높게, faithfulness만 별도로
  낮게 채점되어야 합니다.
- 표현/어휘가 특정 참조 문장과 다르다는 이유로 감점하지 마세요 — 같은 내용을 다른 단어로
  전달해도 무방합니다. 오직 "질문이 요구하는 정보를 실제로 전달하는가"만 보세요.
- 답변이 "질의를 이해하지 못했다"는 취지로 안내만 하고 실제 정보를 전혀 제공하지 않는다면,
  질문이 이 시스템(SRM 평가 기준 FAQ)이 답할 만한 내용으로 보이는데도 회피했다면 낮게
  채점하세요.

**출력 형식**: 다른 설명 없이, 0.0에서 1.0 사이의 숫자 하나만 출력하세요. (예: `0.9`)

## User Prompt 템플릿

```
질문:
{question}

답변:
{answer}

위 답변이 이 질문에 실제로 답하고 있는 정도(relevancy)를 0.0~1.0 사이 숫자 하나로만
출력하세요.
```
